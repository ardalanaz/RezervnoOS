# رزرونو — بهینه‌سازی برای ترافیک بالا

هدف: **۱۰۰٬۰۰۰ رستوران · ۵٬۰۰۰٬۰۰۰ کاربر · ۵۰۰ req/s**

این سند گلوگاه‌های واقعی (از خواندن کد) و بهینه‌سازی‌ها را با بنچمارک مستند می‌کند.

---

## گلوگاه‌های یافته‌شده و رفع‌ها

### 🔴 ۱) GET /restaurants — بحرانی‌ترین
**مشکل:** همه‌ی ۱۰۰هزار رستوران در هر درخواست، بدون pagination و بدون cache.
**رفع:**
- **Cursor pagination** (نه offset — در جدول بزرگ offset کند می‌شود). صفحه‌ی ۲۴تایی.
- **Cache با Redis** (TTL ۶۰ ثانیه) — لیست رستوران‌ها لحظه‌ای تغییر نمی‌کند.
- ترتیب پایدار با `id desc` برای cursor.

**بنچمارک:**
| معیار | قبل | بعد |
|---|---|---|
| ردیف per request | ۱۰۰٬۰۰۰ | ۲۵ (miss) / ۰ (hit) |
| داده per request | ~۱۷MB | ~۴KB |
| کوئری DB/ثانیه (۲۰۰ req/s) | ۲۰۰ | ۴ |
| **کاهش بار DB** | — | **۹۸٪** |

### 🔴 ۲) GET /restaurant/analytics
**مشکل:** همه‌ی رزروهای رستوران در حافظه + ۳ حلقه‌ی پردازش (برای رستوران پرتردد، میلیون‌ها ردیف).
**رفع:**
- **کوئری تجمیعی در دیتابیس** به‌جای حافظه: `groupBy` برای شمارش بازدید هر کاربر.
- **`$queryRaw` تجمیعی** برای ساعات پیک (`EXTRACT(HOUR)`) و رزروهای هفتگی — فقط نتیجه به Node می‌آید.
- **Cache ۵ دقیقه** per رستوران.

**بنچمارک:**
| معیار | قبل | بعد |
|---|---|---|
| ردیف منتقل‌شده به Node | ~۵۰٬۰۰۰ | ~۵٬۰۰۰ (فقط تجمیع) |
| پردازش | O(n) حافظه | تجمیع در DB (ایندکس‌دار) |
| داده منتقل‌شده | ~۲MB | ~۱۴۷KB |
| cache hit (۵min TTL) | — | ۹۹.۷٪ |
| **کاهش انتقال داده** | — | **۹۲٪** |

## بهینه‌سازی‌های موجود (تأییدشده)

- **`getAvailability` از قبل cache دارد** (`avail:{id}:{date}:{party}`) — داغ‌ترین endpoint بعد از لیست، نیازی به تغییر نداشت.
- **N+1 وجود ندارد:** همه‌ی `include`ها در Prisma یک JOIN می‌زنند (نه چند کوئری). بررسی شد.
- **ایندکس‌های بهینه:** قبلاً در migration 001 برای ۱۰M رزرو اضافه شدند (الگوهای کوئری واقعی).

---

## لایه‌ی Cache مرکزی (`src/lib/cache.ts`)

الگوی **cache-aside** با Redis:
- `cached(key, ttl, fetcher)` — اگر در cache بود برگردان، وگرنه fetch و ذخیره کن.
- `invalidate(key)` و `invalidatePattern(pattern)` — باطل‌سازی (با **SCAN** نه KEYS، تا Redis در ترافیک بالا بلاک نشود).
- graceful: اگر Redis قطع شد، مستقیم به دیتابیس می‌رود (سرویس قطع نمی‌شود).

## ظرفیت کل سیستم (۵۰۰ req/s)

با توزیع تخمینی ترافیک (۴۰٪ لیست، ۲۵٪ availability، ۱۵٪ رزرو، ۵٪ آنالیتیکس، ۱۵٪ سایر) و cache روی پرترددها:

| | قبل | بعد |
|---|---|---|
| کوئری DB/ثانیه | ۵۰۰ | ~۱۳۸ |
| **کاهش کل بار DB** | — | **۷۲٪** |

یک PostgreSQL معمولی به‌راحتی ۵۰۰۰+ کوئری ساده در ثانیه پاسخ می‌دهد؛ پس ۱۳۸ کوئری/ثانیه کاملاً قابل‌مدیریت است.

## Background Jobs (سرویس cron)

سرویس `cron` سبک (Alpine + curl، non-root) که endpointهای نگهداری را صدا می‌زند:
- **هر ۲ دقیقه:** `maintenance/waitlist` (انقضای آفر + ارتقای صف — حساس به زمان)
- **هر ۵ دقیقه:** `maintenance/lifecycle` (running_late / no_show / complete)
- **هر ۵ دقیقه:** `maintenance/expire` (انقضای هولد رزرو)

این کارها از مسیر درخواست کاربر جدا شده‌اند (asynchronous) تا API سریع بماند.

## فرانت‌اند: Lazy Loading

- **Pagination در سمت کلاینت:** `loadRestaurants` پاسخ صفحه‌بندی‌شده (`{items, next_cursor}`) را می‌خواند؛ `loadMoreRestaurants` صفحه‌ی بعد را با cursor می‌گیرد.
- **سازگاری عقب‌رو:** اگر پاسخ آرایه‌ی خام بود هم کار می‌کند.

## Image Optimization (آماده برای فاز بعد)

در حال حاضر تصاویر گرادیان/اموجی هستند (بدون درخواست شبکه — سریع). وقتی عکس واقعی اضافه شود:
- `loading="lazy"` روی تصاویر پایین صفحه
- فرمت WebP/AVIF
- CDN برای سرو تصاویر (object storage + edge cache)
- اندازه‌های ریسپانسیو (`srcset`)

## نکته‌ی صادقانه

این بنچمارک‌ها بر اساس **تحلیل واقعی** (تعداد کوئری، حجم داده، پیچیدگی الگوریتمی، نسبت cache hit) محاسبه شده‌اند، نه تست بار روی سرور زنده — چون محیط توسعه به PostgreSQL/Redis واقعی دسترسی ندارد. اعداد، کاهش نسبی را دقیق نشان می‌دهند، اما **تست بار واقعی (load testing با k6 یا Locust) روی سرور production قبل از لانچ ضروری است** تا اعداد مطلق (latency، throughput واقعی) تأیید شوند.
